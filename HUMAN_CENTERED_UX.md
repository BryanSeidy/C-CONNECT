# HUMAN-CENTERED UX MANIFEST

## DESIGN PHILOSOPHY

This application is designed primarily for real people—not for developers.

Many users may:

- have limited digital literacy;
- use smartphones as their only computing device;
- have unstable internet connectivity;
- be unfamiliar with e-commerce platforms;
- have little experience with complex interfaces;
- rely heavily on WhatsApp and Mobile Money.

The interface must therefore reduce cognitive effort as much as possible.

The goal is not to impress.

The goal is to make every task obvious.

Whenever a design decision increases complexity, redesign it.

---

# GOLDEN RULE

If a first-time user needs someone to explain the interface,

the interface has failed.

Every important action should be self-explanatory.

---

# USER PROFILE

Always design for:

- farmers
- producers
- artisans
- cooperatives
- small local businesses
- women entrepreneurs
- first-time digital sellers

Do not assume previous experience with online marketplaces.

---

# MOBILE FIRST

The smartphone is the primary device.

Every screen must be optimized for mobile before desktop.

Design for one-handed usage whenever possible.

Primary actions should be reachable by the thumb.

Avoid horizontal scrolling.

Avoid tiny touch targets.

---

# COGNITIVE LOAD

Reduce thinking.

Reduce choices.

Reduce reading.

Reduce configuration.

Reduce steps.

Reduce surprises.

The interface should guide users naturally.

---

# SIMPLICITY

Each page should answer only one question:

"What does the user need to do here?"

If multiple objectives exist,

split the page.

Avoid multifunction screens.

---

# ONE PRIMARY ACTION

Every screen should contain one obvious primary action.

Users should never hesitate about what to do next.

Primary buttons must be visually dominant.

Secondary actions must remain discreet.

---

# PROGRESSIVE DISCLOSURE

Do not overwhelm users.

Reveal complexity only when needed.

Advanced options should remain hidden until required.

Beginners should never feel intimidated.

---

# FORMS

Forms must be extremely simple.

Split long forms into multiple steps.

Use progress indicators.

Group related information.

Provide examples.

Avoid technical vocabulary.

Validate in real time.

Preserve entered information.

Explain errors clearly.

---

# LANGUAGE

Use plain language.

Avoid technical terms.

Avoid business jargon.

Avoid abbreviations.

Avoid English words when unnecessary.

Every sentence should be understandable by someone unfamiliar with technology.

---

# VISUAL COMMUNICATION

Prefer icons, illustrations and layouts over long explanations.

Show before explaining.

Users should recognize actions visually.

Icons should always accompany important actions.

---

# TRUST

Trust is essential.

Display:

secure payments

verified sellers

clear pricing

delivery status

customer reviews

order history

support availability

Never create uncertainty.

---

# ERROR PREVENTION

Prevent mistakes instead of correcting them later.

Disable impossible actions.

Confirm destructive actions.

Provide clear warnings.

Allow undo whenever possible.

---

# FEEDBACK

Every interaction must generate immediate feedback.

Users should always know:

what happened

what is happening

what will happen next

Silence creates uncertainty.

---

# LOADING STATES

Never leave blank screens.

Always communicate progress.

If an operation takes time,

explain why.

---

# EMPTY STATES

Empty pages should educate users.

Explain:

why the page is empty

what to do next

how to obtain results

Always provide an obvious next step.

---

# NAVIGATION

Navigation must remain predictable.

Maximum 5 primary items.

Avoid deep navigation hierarchies.

Users should never get lost.

Always provide an easy way back.

---

# DASHBOARD

The dashboard should answer:

What happened?

What needs my attention?

What should I do next?

Nothing more.

Avoid information overload.

---

# PRODUCT LISTINGS

Product cards should prioritize:

Image

Name

Price

Availability

Seller

Trust indicators

Everything else is secondary.

---

# ACCESSIBILITY

Support users with:

limited vision

limited literacy

older devices

small screens

poor internet

large fingers

Use large touch targets.

Readable typography.

High contrast.

Simple layouts.

---

# CONNECTIVITY

Assume unstable internet.

Cache aggressively.

Retry automatically.

Preserve user input.

Support offline recovery whenever possible.

Never lose entered information.

---

# PERFORMANCE

Users should feel the application is fast.

Optimize:

images

network requests

bundle size

loading sequences

skeleton screens

optimistic updates

Every second matters.

---

# MICROCOPY

Buttons should use clear verbs.

Examples:

Create Account

Save Product

Continue

Pay Now

Publish

Never use vague labels.

---

# CONFIRMATIONS

Always reassure users.

Examples:

Your product has been published.

Your payment has been secured.

Your order has been confirmed.

Avoid ambiguous success messages.

---

# ONBOARDING

Never assume users understand the application.

Guide them gradually.

Explain features at the moment they are needed.

Avoid long tutorials.

Learning should happen naturally.

---

# DECISION RULE

Whenever multiple UI solutions exist,

choose the one requiring the least mental effort.

Always optimize for clarity.

Never optimize for visual complexity.

---

# FINAL UX OBJECTIVE

The ideal user should be able to:

discover the application,

create an account,

publish a product,

receive an order,

accept payment,

track delivery,

and complete a sale,

without external assistance.

If assistance is required,

the interface must be simplified further.

# FAMILIARITY PRINCIPLE

Whenever possible, reuse interaction patterns that users already know.

Do not reinvent common interactions.

Users should immediately recognize:

buttons

forms

payment flows

navigation

search

product cards

checkout

The interface should feel familiar from the first use.

Learning time should approach zero.

A familiar interface is more valuable than an original one.
